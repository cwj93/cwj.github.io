vue中hash模式与history模式的区别

vue路由的两种模式核心都是-改变视图的同时不向服务发送请求

hash 模式

hash模式的的url中会带有#号，#号虽然出现在url中，但是在发送http请求时，不会被包括在http请求中，对后端没有任何影响。因此改变hash不会重新加载页面。
会使用到的api：

  • hash改变会触发hashchange事件

存在的问题：

  • hash的传参是基于url的，如果要传递复杂的数据，会有体积的限制
  • hash 本来是拿来做页面定位的，如果拿来做路由的话，原来的锚点功能就不能用了

    history 模式

    history模式的URL中不带有#号，该模式是使用pushState,replaceState来修改URL的，会触发popstate事件，我们只需要对popstate事件进行监听获取location.pathname,根据对应的参数修改页面的内容即可。

会使用到的api：

  • pushState,replaceState

存在的问题：

  • history模式需要后端配合。因为在刷新页面之后，浏览器会发送HTTP请求，如果后端没有匹配该Url的话，就会返回404.

总结

1 hash 模式下，仅 hash 符号之前的内容会被包含在请求中，如 http://www.abc.com，因此对于后端来说，即使没有做到对路由的全覆盖，也不会返回 404 错误。

2 history 模式下，前端的 URL 必须和实际向后端发起请求的 URL 一致，如 http://www.abc.com/book/id。如果后端缺少对 /book/id 的路由处理，将返回 404 错误。Vue-Router 官网里如此描述：“不过这种模式要玩好，还需要后台配置支持……所以呢，你要在服务端增加一个覆盖所有情况的候选资源：如果 URL 匹配不到任何静态资源，则应该返回同一个 index.html 页面，这个页面就是你 app 依赖的页面。”

3 结合自身例子，对于一般的 Vue + Vue-Router + Webpack + XXX 形式的 Web 开发场景，用 history 模式即可，只需在后端（Apache 或 Nginx）进行简单的路由配置，同时搭配前端路由的 404 页面支持。

ES6中的数组以及对象的方法

ES6 对象

es6对象中有两个新的方法，分别是Object.assign(),以及Object.is();

  • Object.assign(target, source_1, ···)

    • 用于将源对象的所有可枚举属性复制到目标对象中。
    • null 和 undefined 不能转化为对象，所以会报错
    • assign 的属性拷贝是浅拷贝
    • 数组的处理: 会将数组处理成对象，所以先将 [2,3] 转为 {0:2,1:3} ，然后再进行属性复制，所以源对象的 0 号属性覆盖了目标对象的 0。

  • Object.is(value1, value2)

    • 用来比较两个值是否严格相等，与（===）基本类似。
    • 基本用法 //一是+0不等于-0
      Object.is(+0,-0); //false
      +0 === -0 //true
      //二是NaN等于本身
      Object.is(NaN,NaN); //true
      NaN === NaN //false console.log(Array.of(1, 2, 3, 4)); // [1, 2, 3, 4]

vue事件绑定原理

发布订阅模式

发布-订阅模式其实是一种对象间一对多的依赖关系，当一个对象的状态发生改变时，所有依赖于它的对象都将得到状态改变的通知

既然一对多 肯定有一个事件调度中心用来调度事件 订阅者可以注册事件（on）到事件中心 发布者可以发布事件（emit）到调度中心 订阅者也可以取消订阅（off）或者只订阅一次（once）

主要功能：

  • 维护一个变量 listeners，初识状态为一个空对象字面量，用于存储不同事件和事件对应的处理函数。
  • 实现$on方法：用于监听事件。
  • 实现$once方法：用于一次性事件监听。
  • 实现$emit方法：用于触发对应的事件监听。
  • 实现$off方法：用于消除对应的事件监听。

vue-router钩子函数

路由守卫分类

全局守卫

路由实例上直接操作的钩子函数，特点是所有路由配置的组件都会触发，直白点就是触发路由就会触发这些钩子函数;


路由独享守卫

直接可以在路由配置上设置的钩子函数

组件守卫

在组件内执行的钩子函数,类似于组件内的生命周期，相当于为配置路由的组件添加的生命周期钩子函数。

完整的导航解析流程

  • 1.触发进入其它路由
  • 2.调用要离开路由的组件守卫beforeRouteLeave
  • 3.调用全局的前置守卫beforeEach
  • 4.在重用的组件里调用 beforeRouteUpdate
  • 5.在路由配置里调用 beforeEnter
  • 6.解析异步路由组件
  • 7.在将要进入的路由组件中调用beforeRouteEnter
  • 8.调用全局的解析守卫beforeResolve
  • 9.导航被确认
  • 10.调用全局的后置钩子afterEach。
  • 11.触发 DOM 更新mounted。
  • 12执行beforeRouteEnter守卫中传给 next的回调函数。

new的实现思路

  • 定义一个myNew函数，第一个为构造函数，其他为解构的参数
  • 首先是创建一个实例对象 let obj = {}
  • 接着生成的实例对象 继承 构造函数的原型 obj.proto = fn.prototype
  • 接着改变构造函数的this指向为实例对象 let result = fn.call(obj, …args)
  • 如果构造函数fn的执行结果（result）返回的是一个对象或者函数，那么就直接返回这个对象或函数（result）
  • 如果不是上面的条件，则直接返回实例对象obj

vue生命周期

Posted on 2021-05-27 Edited on 2021-06-08

vue生命周期

Vue 实例有一个完整的生命周期，也就是从开始创建、初始化数据、编译模版、挂载 Dom -> 渲染、更新 -> 渲染、卸载等一系列过程，我们称这是 Vue 的生命周期。

Vue 的父组件和子组件生命周期钩子函数执行顺序？

简述：即在父组件挂载，更新，销毁之前，是子组件先挂载，更新，销毁，再到父组件的挂载，更新，销毁

加载渲染过程:
父 beforeCreate -> 父 created -> 父 beforeMount -> 子 beforeCreate -> 子 created -> 子 beforeMount -> 子 mounted -> 父 mounted

子组件更新过程:
父 beforeUpdate -> 子 beforeUpdate -> 子 updated -> 父 updated

父组件更新过程:
父 beforeUpdate -> 父 updated

销毁过程:
父 beforeDestroy -> 子 beforeDestroy -> 子 destroyed -> 父 destroyed

父组件监听子组件的生命周期

有两种方式：

  • 一种是通过手动$emit触发父组件的事件
  • 第二种是通过@hook:mounted的形式监听子组件的生命周期

    各个生命周期的作用

  • beforeCreate: 组件实例被创建之初，组件的属性生效之前
  • created: 组件实例已经完全创建，属性也绑定，但真实 dom 还没有生成，$el 还不可用
  • beforeMount: 在挂载开始之前被调用：相关的 render 函数首次被调用
  • mounted: el 被新创建的 vm.$el 替换，并挂载到实例上去之后调用该钩子
  • beforeUpdate: 组件数据更新之前调用，发生在虚拟 DOM 打补丁之前
  • update: 组件数据更新之后
  • activited: keep-alive 专属，组件被激活时调用
  • deactivated: keep-alive 专属，组件被销毁时调用
  • beforeDestory: 组件销毁前调用 (清除watcher，子组件事件监听器)
  • destoryed: 组件销毁后调用

SPA单页应用与MPA多页应用

SPA单页应用

只有一个WEB主页面的应用，公共资源(js、css等)仅需加载一次，所有的内容都包含在主页面，对每一个功能模块组件化。单页应用跳转，就是切换相关组件，仅刷新局部资源

关键词：网站模型 、只加载单个HTML页面 、利用路由实现内容变换

描述：SPA（singgle-page application）,单页应用。是一种网站模型，是加载单个HTML页面，并在用户与应用程序交互时动态更新页面。只会在初始化的时候加载相应的HTML,CSS,JS。不会因为用户的操作而进行页面的重新加载或跳转。利用路由机制实现 HTML 内容的变换，UI 与用户的交互，避免页面的重新加载。

  • 传统页面：

    客户端 ———通过发送url请求—————— server

    客户端 ———服务器端返回html页面——— server

  • SPA页面：

    客户端 ———通过发送ajax请求—————— server

    客户端 ———服务器端返回数据——— server

优点：

  • 有良好的交互体验：提升页面交互体验，不会频繁切换浏览器页面，避免了页面的频繁重新加载
  • 前后端分离：前后端职责分离，架构清晰，前端进行交互逻辑，后端负责数据处理
  • 减轻服务器压力：服务器端只需要返回数据即可，不用管展示逻辑与页面合成
    缺点：
  • SEO（搜索引擎优化）难度较高：页面内容动态替换，页面不会刷新，导致很难被搜索引擎抓取（搜索引擎遵循http协议，#不是协议的内容）
  • 无法使用浏览器的前进后退功能，所有页面需要自己建立堆栈管理。（解决方案：利用URI中的散列+iframe实现）（配置好路由信息，通过记录浏览过的历史路由信息，可以很好的记录或历史查看过的界面）
  • 初次加载耗时多：为实现单页 Web 应用功能及显示效果，需要在加载页面的时候将 JavaScript、CSS 统一加载，部分页面按需加载

MPA多页应用

关键词：

描述：MAP（Multi-page application），多页面应用。每访问一个页面都需要重新加载HTML,JS,CSS文件，公共文件按需加载

优点：

  • 首屏加载快
  • SEO（搜索引擎优化）效果好
    缺点：
  • 页面切换慢：每一次切换页面都要重新加载html,css,js等资源
