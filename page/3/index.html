1.1 Vnode

Vnode也称虚拟node节点,是对真实元素节点的抽象。“虚拟 DOM”是我们对由 Vue 组件树建立起来的整个 VNode 树的称呼


createElment函数接受三个参数，分别是：

  • 参数一：标签名、组件的选项对象、函数等等（必选）；

  • 参数二：设置这个对象的样式、属性、传的组件+的参数、绑定事件等（可选）；

  • 参数三：该节点下的其他节点，即子级虚拟节点，可以是字符串形式或数组形式，也需要使用createElement构建。

    Vnode如何检测变化并更新视图呢？

    vue是通过数据绑定来修改视图的，当某个数据发生变化时，会触发Object.defineProperty监听属性的set方法，set方法会调用Dep收集器中的通知方法notify,去通知订阅者watcher，然后订阅者warcher通过get方法调用update方法更新视图。

    更新视图diff算法：不直接对真实DOM操作。将真实DOM抽象成js对象虚拟DOM,然后操作js对象比较新旧对象的差异，经过diff算法得出一些需要修改的最小单位，再将这些小单位的视图进行更新

    diff算法是通过同层的树节点进行比较而非对树进行逐层搜索遍历的方式

1.2 nextTick

任务：
分为同步任务和异步任务

事件循环：
就是从script整体代码（宏任务）开始，遇到同步代码就进入主线程中执行，遇到异步代码则分发任务，setTimeout等会进入下一事件循环宏任务队列中，Promise.then()则会进入当前的事件循环的微任务队列，当前事件循环宏任务执行完毕，接着执行当前事件循环的微任务队列，然后微任务队列执行完毕之后，进入下一事件循环执行宏任务队列。

事件循环的执行顺序为：

宏任务 -> 微任务队列 -> 下一事件循环的宏任务队列(setTimeout) -> 下一事件循环的微任务队列

nextTick的目的是将回调函数的调用延迟到下一次DOM更新数据之后。在vue中，修改数据之后并不会立即更新DOM,因为DOM的更新是异步的，无法通过同步代码获取，需要使用nextTick，在下一次事件循环中获取（Vue中Dom的更新是异步的）

1.3 MVVM

MVC：

  • Model + View + Controller
  • Model: 数据层(模型)，负责数据的处理和获取的数据接口层(是应用程序中用于处理应用程序数据逻辑的部分。通常模型对象负责在数据库中存取数据)
  • View: 视图层，是应用程序中处理数据显示的部分,通常视图是依据模型数据创建的
  • Controller: 控制器层(应用程序中处理用户交互的部分)，通常控制器负责从视图读取数据，控制用户输入，并向模型发送数据
  • MVC 的思想：一句话描述就是 Controller 负责将 Model 的数据用 View 显示出来，换句话说就是在 Controller 里面把 Model 的数据赋值给 View。

MVVM:

  • Model + View + ViewModel
  • Model: 数据层(模型)，负责数据的处理和获取的数据接口层（Model 是指数据模型，泛指后端进行的各种业务逻辑处理和数据操控，对于前端来说就是后端提供的 api 接口）
  • View: 视图层，负责给用户提供操作界面
  • ViewModel: 数据视图层(实现了数据的双向绑定，实现的方式是：DOM 事件监听)，负责处理View层的业务逻辑，做好绑定属性的监听，当ViewModel数据发生变化，View层就会得到更新。
  • 优点：
    • 实现了 View 和 Model 的自动同步
    • 分离View层与Model，降低代码耦合，提高视图或者逻辑的重用性
    • 自动更新DOM:利用双向数据绑定，数据更新后视图更新，减少手动操作DOM次数
    • 只需要关注数据的变化，无需关注DOM操作
  • 缺点：
    • 可能比较难调试。vue数据绑定的声明是指令式写在vue模板中的，无法debugger调试
    • 严格的 MVVM 要求 View 不能和 Model 直接通信，而 Vue 提供了$refs 这个属性，让 Model 可以直接操作 View，违反了这一规定，所以说 Vue 没有完全遵循 MVVM

防抖与节流

Posted on 2021-04-29 Edited on 2021-04-30

防抖与节流

防抖

在触发事件后，等待一段时间后执行事件，如果在此等待时间内，再次触发事件，重置等待时间，再次等待一段时间后执行事件

节流(两种方式，一种是立即执行事件，另一种是延迟执行)

在触发事件之后，立马执行事件，然后在一段时间之内再次触发事件，不会执行事件，只有等待一段时间之后触发事件，才会执行事件


在触发事件时，设置定时器在一段时间之后再执行事件，在此段时间内，多次点击是不会执行事件的。

垃圾回收机制

Posted on 2021-04-28

浏览器的垃圾回收机制

释放垃圾占用的空间，防止内存泄露

引用计数

  • 给每一个对象添加一个计数器，当该对象被引用的时，计数器+1，当引用失效是，计数器-1.用计数器是否为0来判断该对象是否可被回收。
  • 缺点：无法解决循环引用的问题。当两个对象互相引用时，这两个对象的计数器永远不为0

标记清除

  • 先把内存中的对象进行标记，然后将标记为可回收的对象清理掉（有不同的标记状态。存活、可回收、未使用）
  • 缺点：碎片化（直接是在可回收对象的内存区域进行清除，然后会变成未使用的区域），可用新生代/老年代的算法来解决

大文件上传以及断点续传

大文件上传

思路：大文件上传的基本思路是通过切片上传来实现，使用blod.prototype.slice来切割上传的文件，功能跟数组的slice相似。

  • 1.预设好切片的最大数量，使用slice()来切片，返回源文件的某一段的切片
  • 2.借助http请求的并发性，同时上传多个切片（Promise.all），减少上传时间
  • 3.由于是并发，上传的顺序可能发生变化，需要给每个切片记录一下顺序（传递hash参数，使用文件名+数组的形式记录每一个切片的顺序），方便后端合并
  • 4.上传之后还要给后端发送一个合并的请求，此处与切片上传是同步的，是需要文件上传完成之后才发送此请求

Promise的知识点

记录一下Promise的一些知识点

1基础概念

  • Promise是异步编程的一种解决方案，可以将异步操作以同步操作的流程表达出来，避免了层层嵌套的回调函数(回调地狱)
  • Promise有三个状态，分别为pending(等待状态)/fulfilled(成功态)/rejected(失败态)
  • pending状态可以转化为fulfilled/rejected状态
  • 成功时，不可以转为其他状态，且必须有一个不可改变的值(value)
  • 失败时，不可转变为其他状态，且必须有一个不可改变的原因(reason)

任务

  • 任务分为同步任务和异步任务。其中同步任务一般是直接进去主线程中执行的，异步任务是在通过任务队列（先进先出）来协调进入主线程。异步任务包括ajax请求，setTimeout等。

    任务队列

  • 任务队列分为宏任务队列和微任务队列。script( 整体代码)作为一个宏任务，遇到同步代码就直接进入主线程执行，遇到异步代码就分发到对应的任务队列，其中setTimeout等会分发到宏任务队列中，Promise.then()则会分发到微任务队列中。待同步代码（当前宏任务）执行完毕之后，就会执行当前的微任务队列，以先进先出的顺序执行，微任务执行完毕，那么此时就是一个事件循环完成。接着就会进入下一事件循环，接着执行宏任务到微任务的过程。
  • 一个事件循环：宏任务到微任务的执行过程
  • process.nextTick与Promise的then方法在同一事件循环中，process.nextTick的优先级高，先执行

    宏任务与微任务

    • 宏任务：
      • setTimeout
      • setInterval
      • js主代码
      • setImmediate(Node)
      • requestAnimationFrame(浏览器)
    • 微任务：
      • process.nextTick
      • Promise的then方法

href与src的区别

  • href 用于在当前文档和引用资源之间确立联系
    • href 目的不是为了引用资源，而是为了建立联系，让当前标签能够链接到目标地址
    • 引用
    • Hypertext Reference的缩写
    • 指向网络资源所在位置，建立和当前元素（锚点）或当前文档（链接）之间的链接
    • 那么浏览器会识别该文档为css文件，就会并行下载资源并且不会停止对当前文档的处理。这也是为什么建议使用link方式来加载css，而不是使用@import方式
  • src 用于替换当前元素
    • source（缩写），指向外部资源的位置，指向的内容将会应用到文档中当前标签所在位置。
    • 引入
    • source的缩写，指向外部资源的位置
    • 在请求src资源时会将其指向的资源下载并应用到文档内，例如js脚本，img图片和frame等元素
    • 浏览器解析到该元素时，会暂停其他资源的下载和处理，直到将该资源加载、编译、执行完毕，这也是为什么将js脚本放在底部而不是头部

webpack

整理一下平时项目中需要用到的webpack相关的一些优化.

webpack描述

webpack系列之一总览:[https://juejin.cn/post/6844903726981840904]

webpack将每一个文件当成一个模块，经过一系列的处理，为我们整合出最后我们需要的js/css/图片/字体等文件。打包之后的文件里面，是一个自闭自执行函数，参数是 modules,打包的每一个文件都会变编译成module函数，key是文件的相对路径

webpack运行流程

webpack的运行流程是一个【串行】的过程，他的工作流程就是将各个插件【串联】起来

从启动到结束会依次执行一下三个步骤：

  • 初始化流程：从配置文件和shell语句中读取与合并参数，并初始化需要使用的插件和配置插件等执行环境需要的参数
  • 编译构建流程：从Entry(入口)发出，针对每一个Module串行调用对应的Loader去翻译文件内容，再找到该Module依赖的的Module,递归地进行编译处理
  • 输出流程：对编译后的Module组合成Chunk(块),把Chunk转换成文件，输出到文件系统

webpack打包原理

  • 读取webpack配置文件：确定配置文件中的入口文件(./main.js)，获取入口文件的内容
  • 文件位置的解析：从入口文件顺藤摸瓜找到所有文件，递归读取每个模块中的文件内容(代码)
  • 将源码生成AST语法树(抽象语法树),遍历AST语法树，对其转义成浏览器可以运行的代码
  • 将转义后的AST语法树转化成代码并写入配置中的output文件
